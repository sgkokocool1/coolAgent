# 第 16 章：Session、Interrupt、Context 与多轮状态机

文本聊天可以简单地把 messages 列表发给模型；语音 Agent 不行。语音交互是实时的、可打断的、会跨多个设备状态。要做出稳定体验，必须显式管理对话状态。

## 1. 为什么需要状态机

语音 Agent 同时处理四类事件：

- 音频输入：用户开始说话、结束说话、插话。
- 模型输出：LLM partial、工具调用、最终回复。
- 播放输出：TTS 首包、播放中、播放完成、取消。
- 系统事件：超时、网络失败、设备断开、权限拒绝。

如果所有事件都写在一个回调里，很快会出现竞态：旧回复还在播报，新用户语音已经进来；工具还没返回，用户又补充条件。状态机让每个事件在明确状态下被处理。

## 2. 基础状态

一个最小语音助手可以从这些状态开始：

| 状态 | 含义 | 常见下一步 |
| --- | --- | --- |
| `IDLE` | 空闲，等待唤醒 | `LISTENING` |
| `LISTENING` | 正在收音和 VAD | `THINKING`、`IDLE` |
| `THINKING` | ASR 完成，Agent 推理或调用工具 | `SPEAKING`、`LISTENING` |
| `SPEAKING` | TTS 播报中 | `IDLE`、`INTERRUPTED` |
| `INTERRUPTED` | 用户打断旧播报 | `LISTENING`、`THINKING` |
| `ERROR` | 某模块失败 | `IDLE`、`LISTENING` |

`projects/04-voice-assistant/main.py` 演示了这些状态的简化版本。

## 3. Session：一次会话的边界

Session 是对话的容器。它通常包含：

- `session_id`：唯一标识。
- 用户身份和设备信息。
- 当前状态。
- 最近 N 轮消息。
- 正在执行的工具调用。
- 正在播放的 TTS 任务。
- 超时和取消令牌。

Session 的边界取决于产品：

- 智能音箱：一次唤醒到回到空闲。
- 客服电话：整通电话。
- 浏览器助手：一个页面会话。
- 企业工作流：一次任务处理过程。

不要把所有历史都塞进 LLM。Session 保存完整事件，Context Builder 决定本轮给模型哪些内容。

## 4. Interrupt：打断不是异常

打断是正常交互，不应当作为错误处理。用户打断时系统要做：

1. 停止当前 TTS 播放。
2. 标记旧 LLM 输出为 cancelled。
3. 记录用户打断时间。
4. 决定是否保留旧上下文。
5. 处理新 ASR 文本。

常见策略：

- **覆盖**：新意图替代旧意图，适合“等等，改成上海”。
- **追加**：新信息补充旧任务，适合“再加一个明天下午”。
- **确认**：高风险动作被打断后询问用户是否继续。

打断处理的关键是取消令牌。LLM、工具、TTS 都要能感知 session 已取消旧任务，否则后台还会继续消耗资源或产生旧回复。

## 5. Context：给模型什么上下文

Context 不是历史日志的简单拼接。语音 Agent 的上下文通常包括：

- 最近用户话语和系统回复。
- 当前任务槽位，例如城市、日期、联系人。
- 工具调用结果摘要。
- 用户偏好，例如常用城市、语速。
- 系统状态，例如“刚刚被打断”。

示例：

```text
系统：用户刚刚打断了天气播报，新话语可能是在修正旧任务。
历史：用户先问杭州天气。
新输入：等等，改成上海。
```

有了这段上下文，模型才能把“上海”理解成天气查询城市，而不是普通闲聊。

## 6. Multi-turn：多轮任务

多轮对话不是“把上轮文本带上”这么简单。它包含任务槽位的逐步填充：

```text
用户：帮我订明天的票。
助手：去哪里？
用户：上海。
助手：几点出发？
用户：下午三点以后。
```

状态机需要知道当前任务是“订票”，已知槽位是日期和目的地，缺失槽位是出发时间。LLM 可以负责抽取槽位，但业务状态应由确定性代码保存。

## 7. 超时和恢复

语音系统必须处理沉默：

- 用户唤醒后不说话：提示一次，再回到空闲。
- Agent 追问后用户沉默：保留 session 一段时间。
- 工具调用超时：告知用户稍后重试或切换人工。
- TTS 播放失败：显示文本或重试播报。

每个超时都应写在状态机中，而不是散落在 SDK 回调里。

## 8. 日志和可观测性

建议记录事件流：

```json
{"state":"LISTENING","event":"start_of_speech","time_ms":1200}
{"state":"THINKING","event":"asr_final","text":"查一下杭州天气"}
{"state":"SPEAKING","event":"tts_first_chunk","latency_ms":430}
{"state":"INTERRUPTED","event":"barge_in","time_ms":2600}
```

这些日志能回答关键问题：

- 用户等了多久才听到首包？
- 是 VAD 截断还是 ASR 识别错？
- 打断是否及时停止了 TTS？
- 工具调用是否拖慢整体体验？

## 9. 简化实现模式

可以用一个纯 Python 状态机开始：

```python
class VoiceSession:
    def on_event(self, event):
        if self.state == "SPEAKING" and event.type == "USER_SPEECH":
            self.cancel_tts()
            self.state = "INTERRUPTED"
```

随着系统变复杂，再引入队列、Actor 或工作流引擎。无论技术形态如何，状态、事件和转移都要清晰。

## 10. 验收清单

一个合格的语音对话状态模块应满足：

- 所有状态有明确进入和退出条件。
- 打断可以取消旧 TTS 和旧推理。
- Context Builder 能区分历史日志和本轮模型上下文。
- 多轮任务槽位由业务状态保存。
- 超时、失败和权限拒绝都有用户可理解的回复。
- 状态迁移可记录、可测试、可复盘。
