# 第 13 章：语音 Agent 完整链路

语音 Agent 的本质不是“给聊天机器人加一个麦克风”，而是把实时音频、自然语言理解、工具调用、对话管理和语音合成组织成一条低延迟、可打断、可观测的流水线。本章先建立全局地图：

```text
Mic -> VAD -> ASR -> LLM -> Tool -> LLM -> TTS -> Speaker
```

在真实系统中，每个箭头都可能跨线程、跨进程甚至跨机器；在本教程中，`examples/12-voice-pipeline` 用文本事件模拟同样的数据流，帮助你在没有麦克风和模型的环境里练习架构。

## 1. Mic：音频入口

麦克风入口负责持续采集 PCM 帧，常见格式是 16 kHz、16 bit、单声道。生产系统会在入口层处理：

- 设备选择：默认麦克风、蓝牙耳机、浏览器音频流。
- 回声消除：避免扬声器播放的 TTS 被再次录入。
- 降噪和增益：提升远场或嘈杂环境识别率。
- 帧切分：例如每 10 ms 或 20 ms 产出一帧。

教学示例里，`MockMicrophone` 不采集真实音频，而是产出 `AudioEvent("speech")` 和 `AudioEvent("silence")`。这让后续模块仍然按真实链路工作。

## 2. VAD：判断何时开始和结束

VAD（Voice Activity Detection）回答两个问题：

1. 用户是否正在说话？
2. 这句话是否已经说完？

只要 VAD 没有给出端点，ASR 就不能贸然把半句话提交给 Agent。端点过早会造成“我想订一张去……”被截断；端点过晚会让用户感觉系统迟钝。真实系统常组合短静音、最大语音时长和语义断句来判断结束。

## 3. ASR：把语音转为文本

ASR 接收 VAD 确认的语音片段，也可以接收流式音频并不断输出部分结果。语音 Agent 更偏好流式 ASR，因为它能提前发现用户意图，甚至在用户没说完时准备候选工具。

工程上要关注：

- 语言和热词：人名、产品名、城市名会影响工具参数。
- 时间戳：字幕、打断回放和质量分析都依赖时间戳。
- 断句：逗号、句号会影响 LLM 的上下文理解。
- 置信度：低置信度时可以追问，而不是直接执行高风险工具。

## 4. LLM：理解意图并决定工具

第一轮 LLM 通常做三件事：

1. 识别用户意图。
2. 判断是否需要工具。
3. 生成工具名和参数。

例如“帮我查一下杭州今天的天气”会变成：

```json
{
  "name": "get_weather",
  "arguments": {"city": "杭州", "date": "today"}
}
```

在语音场景中，LLM 还要处理口语化表达、省略和修正。例如用户先说“查一下杭州天气”，又立刻打断“等等，改成上海”，状态管理层必须让新意图覆盖旧播报。

## 5. Tool：让 Agent 接触外部世界

工具层执行确定性动作，例如查天气、建日程、搜文档、发工单。工具调用要遵守三条规则：

- **可审计**：记录调用者、参数、时间和结果。
- **可回滚**：高风险动作需要确认或补偿。
- **可降级**：工具失败时给出明确中文解释。

教学示例的 `MockToolRegistry` 提供天气和提醒工具，返回固定文本，便于离线运行。

## 6. 第二轮 LLM：汇总工具观察

工具返回的是结构化结果或原始文本，不一定适合直接播报。第二轮 LLM 需要把 Observation 变成自然、简短、适合听的回复。语音回复应避免长表格和复杂嵌套，优先给结论，再补充必要细节。

不好：

> 根据接口返回 JSON，字段 temperature_min 为 22，temperature_max 为 27，weather_code 为 308。

更好：

> 杭州今天小雨转多云，22 到 27 度，建议出门带伞。

## 7. TTS：低延迟播报

TTS 把文本合成音频。语音 Agent 关注首包延迟和可打断性。常见做法是：

- LLM 流式生成时，按短句送入 TTS。
- TTS 边合成边播放，不等待全文结束。
- 播放器保留取消接口，用户插话时立即停止。
- 对重要数字、人名和英文缩写做读音规范化。

示例中的 `MockTTS` 用分块打印模拟流式播放，并在第一轮播报中途触发“用户打断”。

## 8. Speaker：输出和回声问题

扬声器不仅是输出设备，还会影响输入。若没有回声消除，系统可能把自己播报的内容重新识别为用户语音。工程上通常需要：

- 播放状态反馈给 VAD/ASR。
- 回声消除或播放参考信号。
- TTS 音量、语速和音色控制。
- 设备断开后的恢复策略。

## 9. 打断：语音 Agent 的关键体验

打断（barge-in）是语音交互区别于文本聊天的重要能力。用户不想等系统说完一大段才纠正。完整流程是：

1. TTS 正在播报。
2. Mic/VAD 检测到新的用户语音。
3. 播放器停止当前音频。
4. ASR 识别新语音。
5. 对话状态机决定覆盖、合并还是追问。
6. Agent 基于新意图继续回复。

`examples/12-voice-pipeline/main.py` 展示了这个过程：第一轮查天气，播报中途被“提醒我带伞”打断，随后进入第二轮。

## 10. 最小可运行练习

运行：

```bash
python3 examples/12-voice-pipeline/main.py
```

观察输出中的这些阶段：

- `[Mic]`：假音频事件。
- `[VAD]`：起点和端点。
- `[ASR]`：文本转写。
- `[Tool]`：函数调用结果。
- `[LLM]`：最终回复。
- `[MockTTS]`：分块播报和打断。

当你把 Mock 模块替换成真实模型时，优先保持接口不变：VAD 输出 utterance，ASR 输出 transcript，Agent 输出 answer，TTS 接收 text 并支持 cancel。
