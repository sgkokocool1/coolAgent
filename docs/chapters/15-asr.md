# 第 15 章：ASR 选型、流式识别与时间戳

ASR（Automatic Speech Recognition）把语音变成文本，是语音 Agent 理解用户意图的入口。对普通转写系统来说，“最终文本准确”很重要；对语音 Agent 来说，还要关注流式延迟、断句质量、时间戳、热词和置信度。

## 1. ASR 的输入输出

输入通常是 VAD 切出的音频片段或持续音频流。输出可以是：

- 最终文本：一句话结束后的完整识别。
- 部分文本：流式识别中的临时结果。
- 词级时间戳：每个词或字的开始结束时间。
- 句级时间戳：每个断句的范围。
- 置信度：识别结果可信程度。

教学示例中的 `MockASR` 直接把 `Utterance.text` 返回为 transcript。虽然没有真实模型，但保留了 `transcribe(utterance) -> str` 这个边界。

## 2. FunASR Nano

FunASR Nano 适合中文场景快速落地，常见优势：

- 中文识别体验较好。
- 支持端点、标点、热词等生态能力。
- 部署方式灵活，可本地或服务化。

适合：

- 中文语音助手。
- 客服质检和会议摘要。
- 需要中文热词的业务系统。

注意点：

- 模型版本和导出格式要固定，避免线上效果漂移。
- 热词要评估副作用，不能让普通词被强行识别成品牌词。
- 若做实时 Agent，需要确认流式接口的部分结果稳定性。

## 3. Sherpa-ONNX

Sherpa-ONNX 面向端侧和跨平台部署，适合把 ASR 放到本地设备：

- 基于 ONNX，便于在 CPU、移动端和嵌入式设备运行。
- 支持多种模型族和流式识别。
- 不依赖云端，隐私和离线能力更好。

适合：

- 桌面助手、车载助手、离线设备。
- 对网络不可用或隐私敏感的场景。
- 需要统一 C++/Python/移动端接口的项目。

注意点：

- 端侧算力决定实时率。
- 模型量化会影响准确率。
- 需要自己处理音频设备、VAD 和线程调度。

## 4. Whisper.cpp

Whisper.cpp 是 Whisper 的高性能 C/C++ 实现，优点是部署简单、跨平台好、社区活跃：

- 可在 CPU 上运行，适合本地 demo。
- 多语言能力强。
- 支持量化模型，降低资源占用。

适合：

- 离线转写。
- 多语言输入。
- 原型验证和桌面工具。

限制：

- 原生 Whisper 更偏离线分段转写，低延迟流式需要额外工程。
- 长音频需要处理上下文窗口和幻觉问题。
- 对实时打断场景，要结合 VAD 和增量策略。

## 5. Qwen3-ASR

Qwen3-ASR 代表新一代大模型 ASR/语音理解方向，通常更强调复杂语音、口语表达和上下文能力。它适合探索：

- 更强的中文和多语言识别。
- 与大语言模型上下文协同。
- 复杂口语、领域词和语义纠错。

工程接入时要确认：

- 是否支持流式。
- 是否返回时间戳。
- 并发、成本和延迟是否符合语音交互。
- 私有化部署和数据合规要求。

## 6. 流式 ASR

流式 ASR 会不断输出 partial result：

```text
用户正在说：帮我查一下杭州今天的天气
partial: 帮我
partial: 帮我查一下
partial: 帮我查一下杭州
final: 帮我查一下杭州今天的天气
```

语音 Agent 可以利用 partial result 做提前准备：

- 检测到“天气”后预热天气工具。
- 检测到“搜索”后建立检索请求。
- 检测到高风险动作时准备确认话术。

但不要过早执行不可逆工具。partial result 可能被后续文本推翻，例如“帮我订票……算了，先查一下价格”。

## 7. 断句与标点

ASR 输出没有标点会影响 LLM。断句模块负责把连续文本切成适合理解的单位：

- 根据静音时长断句。
- 根据语气词和语义完整性断句。
- 根据最大长度强制断句。
- 对数字、日期、金额做规范化。

示例：

```text
原始：明天下午三点提醒我给王总打电话然后查一下上海天气
断句：明天下午三点提醒我给王总打电话。然后查一下上海天气。
```

断句结果可以直接影响工具调用数量：一句话里可能有两个任务。

## 8. 时间戳

时间戳不仅用于字幕，还用于调试和产品体验：

- 用户投诉“系统抢话”时，可以看端点时间。
- TTS 打断时，可以定位用户从哪一刻开始说话。
- 会议摘要可以把结论链接回原始音频。
- 低置信度片段可以高亮给人工校对。

常见粒度：

- 字/词级时间戳：精细但数据多。
- 句级时间戳：适合字幕和摘要。
- utterance 级时间戳：适合 Agent 日志。

## 9. ASR 与上下文

语音 Agent 的 ASR 不应完全无状态。上下文可以提升识别：

- 当前城市、联系人、产品名。
- 用户历史偏好。
- 当前页面或业务流程。
- 最近工具调用结果。

例如用户在差旅系统里说“帮我订到上海的票”，ASR/Agent 可以把“票”理解为“车票或机票”，并结合页面上下文追问。

## 10. 评估指标

除了 WER/CER，还要评估：

- 首个 partial 延迟。
- final 延迟。
- 端点后提交延迟。
- 热词召回率。
- 标点和断句准确率。
- 打断场景误识别率。
- 工具参数抽取成功率。

语音 Agent 的最终目标不是转写漂亮，而是任务完成率高、用户等待短、错误可恢复。

## 11. 接入建议

从 Mock 迁移到真实 ASR 时，保持接口分层：

```python
class ASR:
    def transcribe(self, utterance) -> str:
        ...
```

如果使用流式：

```python
class StreamingASR:
    def accept_audio(self, frame) -> list[PartialResult]:
        ...
```

不要让业务 Agent 直接依赖某个 ASR SDK 的对象。把 SDK 细节包在适配器里，后续才能从 Whisper.cpp 切到 Sherpa-ONNX 或云端 ASR。
