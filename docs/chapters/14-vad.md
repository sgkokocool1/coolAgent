# 第 14 章：VAD、起止判定与端点检测

VAD（Voice Activity Detection）是语音 Agent 的“耳朵开关”。它不理解用户说了什么，只判断当前音频是否像人声，以及一句话是否已经结束。VAD 做得好，系统会显得灵敏、自然；VAD 做得差，ASR 和 LLM 再强也会被错误切分拖累。

## 1. VAD 在链路中的位置

```text
Mic -> VAD -> ASR -> Agent
```

VAD 位于麦克风之后、ASR 之前。它的输出通常包括：

- `start_of_speech`：检测到用户开始说话。
- `speech_frame`：当前帧属于语音。
- `end_of_speech`：检测到一句话结束。
- `no_speech`：静音或噪声。

真实系统会把音频按 10 ms、20 ms 或 30 ms 切帧，每帧给 VAD 打分。教学示例用 `speech` 和 `silence` 文本事件模拟这些帧。

## 2. WebRTC VAD

WebRTC VAD 是经典轻量方案，优点是快、依赖少、CPU 占用低。它适合：

- 浏览器或端侧实时检测。
- 对延迟敏感的唤醒后监听。
- 资源受限设备。

常见限制：

- 对音乐、键盘声、环境噪声较敏感。
- 只能判断是否有人声，不提供语义信息。
- 参数较少，复杂场景需要外部平滑策略。

WebRTC VAD 通常有 aggressiveness 级别，越激进越容易把弱人声判成静音，越保守越容易把噪声判成人声。语音助手一般从中间档开始，再根据误触发率调整。

## 3. Silero VAD

Silero VAD 是神经网络 VAD，常用于 Python 服务端或边缘设备。优点是鲁棒性更好，能给出语音概率。它适合：

- 噪声更复杂的环境。
- 需要更细阈值控制的服务端。
- 离线批处理切分长音频。

常见输出是某帧的 speech probability。你可以设置：

- `threshold`：超过多少概率算语音。
- `min_speech_duration_ms`：最短有效语音时长。
- `min_silence_duration_ms`：端点静音时长。
- `max_speech_duration_s`：最长一句话，防止一直不结束。

## 4. 起点判定

起点判定不应只看单帧。单帧误判会让键盘声、咳嗽声触发 ASR。常见策略：

1. 连续 N 帧语音概率超过阈值。
2. 最近窗口内语音帧比例超过阈值。
3. 音量门限和 VAD 概率同时满足。
4. 唤醒词后短时间内降低触发门槛。

示例：

```text
最近 5 帧中至少 3 帧为 speech -> start_of_speech
```

起点太慢会吞掉开头字，起点太快会增加误触发。端侧系统通常保留一小段 pre-roll buffer，把起点前 200 ms 音频一起送给 ASR，避免“帮我查”变成“查”。

## 5. 终点判定

终点判定也叫 endpointing。它决定什么时候把用户的一句话提交给 ASR/LLM。常见规则：

- 已检测到语音。
- 语音持续超过最短时长。
- 后续静音持续超过阈值。
- 没有超过最大单句时长。

示例：

```text
speech_ms >= 300 且 silence_ms >= 600 -> end_of_speech
```

短命令可以用更短静音，例如 300 ms；长问答可以用 700-1000 ms。中文口语中停顿很多，如果阈值太短，会把“我想订一张……明天去上海的票”切成两句。

## 6. 语义端点检测

仅靠静音无法理解“是否说完”。更高级的系统会加入语义端点检测：

- ASR 部分结果是否形成完整句子。
- 末尾是否是“然后”“还有”“帮我”等未完成结构。
- 用户是否正在列举。
- 当前任务是否需要更多槽位。

例如：

> 用户：帮我订一张明天去

静音 800 ms 后，VAD 可能认为结束，但语义上目的地缺失。状态机应追问“去哪里？”而不是直接执行。

## 7. 打断场景中的 VAD

当 TTS 正在播报时，VAD 的任务更难，因为麦克风会收到扬声器声音。处理方式包括：

- 回声消除：用播放参考信号消除 TTS。
- 播放期间提高 VAD 阈值。
- 要求新语音持续更长时间才算打断。
- 检测到打断后立即停止 TTS，降低回声。

打断策略要平衡：太敏感会被自己的播报打断；太迟钝会让用户无法纠正。

## 8. 本仓库 Mock VAD

`shared/voice_mocks.py` 中的 `MockVAD` 用两个参数模拟真实策略：

- `min_speech_ms`：最短有效语音。
- `endpoint_silence_ms`：端点静音时长。

它接收 `AudioEvent("speech")` 累积文本，接收 `AudioEvent("silence")` 后判断是否输出 `Utterance`。这不是音频算法，但保留了真实系统最重要的接口：只有端点成立，ASR 才会收到完整 utterance。

## 9. 调参建议

从产品体验出发调参，而不是只看模型指标：

| 场景 | 起点策略 | 终点策略 |
| --- | --- | --- |
| 智能音箱 | 保守，避免误唤醒 | 中等，支持自然停顿 |
| 客服电话 | 中等，减少抢话 | 偏长，避免截断客户 |
| 车载助手 | 保守，抗噪声 | 偏短，命令式交互 |
| 会议字幕 | 灵敏，尽量不断流 | 不强制端点，交给断句 |

## 10. 验收清单

一个可用的 VAD/端点模块至少要满足：

- 能输出起点和终点日志。
- 能配置最短语音和端点静音。
- 能处理短噪声，不触发 ASR。
- 能处理用户停顿，不频繁截断。
- 播报期间能支持打断策略。
- 所有决策都能记录到日志，便于复盘。
